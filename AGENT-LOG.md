# Agent Log

Append-only run log for the autonomous dev agent. Every run writes one entry
here, regardless of outcome. Review cycles write a separate `REVIEW-LOG.md`
entry AND flag this entry with `review_proposed: true` — so the log never has
gaps.

## Format

```
---

### Run [YYYY-MM-DD HH:MM]
- Task: TASK-XXX — <title> (or "N/A — no tasks available")
- Outcome: success | blocked | skipped | success_with_warning
- PR: <URL or N/A>
- Test counts: <workspace counts>
- Files changed: <list>
- Regression alert: true | false
- Review proposed: true | false
- Deploy: success | failed | n/a
- Lessons learned: <free text>
- Notes: <optional>
```

Multiplicative would have been tidy in isolation (base × windFactor × currentFactor) but composes poorly with future per-tile cost variations (storms, reefs, shoals) that the engine is likely to add — you want those and wind to stack independently so the tile-type decides "how long to cross" and the zones decide "how the weather helps or hurts," not a fused product that has to re-derive the base every time. The hard 0.25 floor (`MIN_SAILING_STEP_COST`) exists so double-stacked tailwind+current can't push cost to 0 (degenerate for A* — zero-cost steps create loops in the relaxation) and Infinity propagation stays explicit (Red Tide without immunity returns Infinity before offsets are even computed; favourable wind can't "rescue" an impassable tile).
  - Wind and current are *two separate* DirectionLayers, not one merged "weather" layer. They're conceptually distinct: prevailing wind affects sailing ergonomics (short-run + changeable across the map), ocean current is a persistent push (long-run + aligned with the Rayon Passage flow), and downstream mechanics will read them independently — combat fires directly on wind alignment (TASK-053 ship-combat resolver), raiding reads current for chase mechanics (TASK-055), rumour-text generation in STORY-26 references "wind-of-the-sea" as faction flavour per-register. Merging them would require each consumer to re-split on a synthetic "which kind of push" field; keeping them separate keeps the save format and the consumer APIs honest. Generation sizes differ accordingly — 2–4 wind zones of 6–14 cells (more + smaller, patchier weather) vs 1–3 current zones of 4–10 cells (fewer + medium-scale, persistent).
  - A* admissibility under variable step cost: the old heuristic `max(|dx|, |dy|)` was tight-admissible when every step cost exactly 1 (Chebyshev distance = true min-cost). With the min-step-cost now at 0.25, that heuristic *over*estimates the true minimum (a dual tailwind+current path could cost 0.25 per step, not 1), which would let A* miss optima. Fix: scale the heuristic by `MIN_SAILING_STEP_COST` *only when wind or current flags are present* (`flags.wind !== undefined || flags.current !== undefined`) — otherwise the tight baseline heuristic still applies and no-wind searches keep their former pruning efficiency. This is the cheapest-admissible heuristic for the mixed case without re-deriving a per-call lower bound; it trades some pruning tightness for correctness.
  - Zones only seed on navigable water (Ocean / RayonPassage / FloatingCity / FataMorgana) — Islands and Red Tide are excluded. Reasons: (a) an island tile with a tailwind is meaningless to a ship that can't enter it, (b) the blob-grow candidate filter is a lot cleaner when it just asks `isNavigable(type)` rather than maintaining a separate "zone-eligible" allow-list, and (c) zones over Red Tide would interact oddly with the immunity flag — favourable wind on a Red Tide tile for a Kraken-Talisman-carrying unit makes a perverse sort of sense, but the gameplay read ("wind pushes you into the Red Tide faster") is not what the rules want. Red Tide stays a terrain hazard with its own gate. If a future task wants "storm weather" or "Abyssal-gale" modifiers tied to eldritch tiles, that's a new zone type with its own eligibility filter, not a relaxation of this one.
  - Continuing the "ship the primitive; leave the iteration to the collection task" pattern from TASK-023 / TASK-024 / TASK-026 / TASK-034 — the primitive here is `sailingStepCost(map, from, to, flags)` + `findPath(..., { wind, current })`, and the costing is explicitly per-edge rather than baked into a "wind-aware pathfinder" wrapper. When the unit-move resolver (STORY-16 already shipped but will extend) calls `findPath` for a ship order, it passes the generated `wind` and `current` layers from game state; when ground combat resolves later (TASK-053/054) it doesn't pass them — same findPath, different flag shape. The layers live on `GeneratedMap` next to `factionStarts` and `rumours`, which means they're part of the world once but consumed zero-to-many times per turn without re-computation. The save format captures them via `DirectionLayer.toJSON()` (sparse row-major entries, deterministic sort) the same way the visibility module captures per-faction state.
