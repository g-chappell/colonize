# Agent Log

Append-only run log for the autonomous dev agent. Every run writes one entry
here, regardless of outcome. Review cycles write a separate `REVIEW-LOG.md`
entry AND flag this entry with `review_proposed: true` — so the log never has
gaps.

## Format

```
---

### Run [YYYY-MM-DD HH:MM]
- Task: TASK-XXX — <title> (or "N/A — no tasks available")
- Outcome: success | blocked | skipped | success_with_warning
- PR: <URL or N/A>
- Test counts: <workspace counts>
- Files changed: <list>
- Regression alert: true | false
- Review proposed: true | false
- Deploy: success | failed | n/a
- Lessons learned: <free text>
- Notes: <optional>
```

Multiplicative would have been tidy in isolation (base × windFactor × currentFactor) but composes poorly with future per-tile cost variations (storms, reefs, shoals) that the engine is likely to add — you want those and wind to stack independently so the tile-type decides "how long to cross" and the zones decide "how the weather helps or hurts," not a fused product that has to re-derive the base every time. The hard 0.25 floor (`MIN_SAILING_STEP_COST`) exists so double-stacked tailwind+current can't push cost to 0 (degenerate for A* — zero-cost steps create loops in the relaxation) and Infinity propagation stays explicit (Red Tide without immunity returns Infinity before offsets are even computed; favourable wind can't "rescue" an impassable tile).
  - Wind and current are *two separate* DirectionLayers, not one merged "weather" layer. They're conceptually distinct: prevailing wind affects sailing ergonomics (short-run + changeable across the map), ocean current is a persistent push (long-run + aligned with the Rayon Passage flow), and downstream mechanics will read them independently — combat fires directly on wind alignment (TASK-053 ship-combat resolver), raiding reads current for chase mechanics (TASK-055), rumour-text generation in STORY-26 references "wind-of-the-sea" as faction flavour per-register. Merging them would require each consumer to re-split on a synthetic "which kind of push" field; keeping them separate keeps the save format and the consumer APIs honest. Generation sizes differ accordingly — 2–4 wind zones of 6–14 cells (more + smaller, patchier weather) vs 1–3 current zones of 4–10 cells (fewer + medium-scale, persistent).
  - A* admissibility under variable step cost: the old heuristic `max(|dx|, |dy|)` was tight-admissible when every step cost exactly 1 (Chebyshev distance = true min-cost). With the min-step-cost now at 0.25, that heuristic *over*estimates the true minimum (a dual tailwind+current path could cost 0.25 per step, not 1), which would let A* miss optima. Fix: scale the heuristic by `MIN_SAILING_STEP_COST` *only when wind or current flags are present* (`flags.wind !== undefined || flags.current !== undefined`) — otherwise the tight baseline heuristic still applies and no-wind searches keep their former pruning efficiency. This is the cheapest-admissible heuristic for the mixed case without re-deriving a per-call lower bound; it trades some pruning tightness for correctness.
  - Zones only seed on navigable water (Ocean / RayonPassage / FloatingCity / FataMorgana) — Islands and Red Tide are excluded. Reasons: (a) an island tile with a tailwind is meaningless to a ship that can't enter it, (b) the blob-grow candidate filter is a lot cleaner when it just asks `isNavigable(type)` rather than maintaining a separate "zone-eligible" allow-list, and (c) zones over Red Tide would interact oddly with the immunity flag — favourable wind on a Red Tide tile for a Kraken-Talisman-carrying unit makes a perverse sort of sense, but the gameplay read ("wind pushes you into the Red Tide faster") is not what the rules want. Red Tide stays a terrain hazard with its own gate. If a future task wants "storm weather" or "Abyssal-gale" modifiers tied to eldritch tiles, that's a new zone type with its own eligibility filter, not a relaxation of this one.
  - Continuing the "ship the primitive; leave the iteration to the collection task" pattern from TASK-023 / TASK-024 / TASK-026 / TASK-034 — the primitive here is `sailingStepCost(map, from, to, flags)` + `findPath(..., { wind, current })`, and the costing is explicitly per-edge rather than baked into a "wind-aware pathfinder" wrapper. When the unit-move resolver (STORY-16 already shipped but will extend) calls `findPath` for a ship order, it passes the generated `wind` and `current` layers from game state; when ground combat resolves later (TASK-053/054) it doesn't pass them — same findPath, different flag shape. The layers live on `GeneratedMap` next to `factionStarts` and `rumours`, which means they're part of the world once but consumed zero-to-many times per turn without re-computation. The save format captures them via `DirectionLayer.toJSON()` (sparse row-major entries, deterministic sort) the same way the visibility module captures per-faction state.

- Lessons learned:
  - Three-file split inside `packages/core/src/diplomacy/` — `diplomacy-action.ts` (const-object + effect table + aiShouldAccept), `relations-matrix.ts` (data primitive), `attempt.ts` (orchestrator) — rather than one mega-module. Reason: the three files have independent change cadences. Adding a new action is a diplomacy-action.ts-only change (new literal + new switch case + new threshold case — TypeScript exhaustiveness pins the three touch-points). Re-tuning a cooldown or threshold is a data edit in the same file. Adding a new *data shape* (say, per-pair event log, or opinion-modifier stack) is a relations-matrix.ts edit that leaves the action table untouched. Adding new attempt semantics (pre-conditions, emit a side-effect event) is attempt.ts-only. Monolith would have been ~600 lines of heterogeneous concerns; the split mirrors `combat/` (combat-action + combatant + resolve + event).
  - Symmetric pair semantics via a *canonical key* inside `RelationsMatrix` (`a < b ? "a|b" : "b|a"`) rather than storing two directional entries or exposing an ordered-pair API. Reasons over two-entries: halves the storage footprint, eliminates the "what if (a,b) and (b,a) drift" bug class, keeps toJSON output compact. Reasons over ordered-pair API: diplomacy actions are symmetric by the game's rules (declare-war affects both sides, cooldown applies to both), so exposing order at the signature level would either force every consumer to sort themselves or invite inconsistent call-site conventions. Pipe separator is safe because FactionId values are kebab-case identifiers — TILESPAN-style escaping would be premature. Same canonical-key pattern as the `cells` key in `DirectionLayer` but applied to pair identity instead of grid index.
  - Cooldowns stored as *absolute expiry turn* (`expiresOnTurn`), not as a countdown. The matrix has no reference to the TurnManager — consumers pass `currentTurn` on every `isOnCooldown` / `attemptDiplomacyAction` call. Reasons: (a) countdown state would need a per-turn decrement hook registered against the TurnManager, which ties this primitive to the turn scheduler — exactly the coupling CLAUDE.md's "Ship the entity's primitive; leave iteration / scheduling to the task that owns the collection" rule guards against. (b) A save snapshot mid-turn is unambiguous: "cooldown expires on turn 14" reads the same whether you reload during turn 9's AI phase or turn 10's player phase; a countdown integer is interpretable only relative to a per-turn tick that the save format would have to carry. (c) Replay determinism is easier: the test asserts `isOnCooldown(..., currentTurn=10) === false` once `expiresOnTurn=10`, no "advance turn X times then check" wrapper.
  - Separated `shiftScore` (clamped delta) from `setScore` (absolute, bounds-validated). Gameplay code should only ever use `shiftScore` — relations are earned or spent, not dictated. `setScore` exists for the save-load path (fromJSON reads the absolute number directly) and for deterministic test seeding. Keeping them as two surfaces makes the "only use shift in live game code" rule enforceable at code review: any `setScore` call outside `fromJSON` / tests is a red flag. Exposed both because a single method that accepted an "absolute or delta?" second arg would either be confusing or land on a magic convention (delta if `< 100 && > -100`?).
  - Attempt outcome as a *discriminated union* keyed by `status` — `'accepted' | 'declined' | 'blocked-cooldown' | 'invalid-same-faction'` — rather than a single shape with nullable fields. Reasons: `exactOptionalPropertyTypes: true` is enabled (CLAUDE.md) so mixing "has cooldownUntil when blocked / has scoreDelta when not blocked" into one shape means the type has to declare both as optional and every consumer branches on truthiness, which TypeScript doesn't narrow correctly across two conditional fields. Union discriminator narrows cleanly at every call site (`if (outcome.status === 'accepted') outcome.newScore …`) and the invalid-same-faction variant carries *zero* mutation-state fields — it cannot pretend to have a score delta the matrix didn't apply. Cost: four interface declarations instead of one. Benefit: no "did that run mutate or not?" ambiguity at the use site.
  - Action effect table via an exhaustive `switch(action)` with no `default` in `getDiplomacyActionEffect` rather than a `Record<DiplomacyAction, DiplomacyActionEffect>` object literal. The CLAUDE.md "consume save-format const-object unions via an exhaustive switch with no default" rule applies: adding a new `DiplomacyAction` literal causes `getDiplomacyActionEffect` and `aiShouldAccept` to both fail to compile until the author picks concrete numbers and a threshold for the new case. A `Record` would still give a compile error on the writer side (you must add the key) but not on the *reader* side if someone imported the table indirectly — the switch makes every reader a build-time tripwire. Same precedent as `tileCost(TileType)` in `pathfind.ts`.
  - Matrix owns no knowledge of the action table — `RelationsMatrix.shiftScore(a, b, delta)` accepts any integer delta; the attempt orchestrator reads the effect table and supplies the right delta. This mirrors the "scalar seam" pattern (CLAUDE.md): the data primitive doesn't look up policy, the orchestrator passes policy through. Benefits: re-tuning action numbers is a data edit in diplomacy-action.ts with zero matrix-test churn, an alternate policy layer (e.g. AI personality modifier that scales deltas by 0.8x) slots in at the orchestrator without the matrix knowing, and tests for `shiftScore` stay at the primitive level (delta clamping, determinism) while tests for action numbers stay at the action-table level.

  - Split the task into two orthogonal files inside `packages/core/src/chimes/` — `chimes-registry.ts` for the per-building rate + threshold ladder + pure summation helpers, and `chimes-ledger.ts` for the per-faction mutable primitive (accumulator + threshold-crossing detection + deferred-event queue + save-format round-trip). Benefit: adding a new chime-producing building is a pure registry edit (one entry in `CHIMES_PER_TURN_BY_BUILDING`, one test case) with zero ledger-test churn; rewiring the event-queue semantics (say, from FIFO to per-faction-priority) is a ledger-file edit with zero registry-test churn.
  - Shipped *only* the primitive — no `TurnManager` hook, no per-colony iteration, no per-faction ledger map. The task description's "Chimes produced per turn per qualifying building" reads like an orchestration spec, but per CLAUDE.md's "Ship the entity's primitive; leave iteration / scheduling to the task that owns the collection" rule the iteration belongs to whichever future task owns the colony collection + per-faction ledger map. The ledger's single-faction shape makes that wrapping trivial (`Map<PlayableFactionId, ChimesLedger>`) without forcing this task to pick a Turn phase (Start? End? WorldEvents? depends on whether population-growth / tithe-drain wants to fire before or after the chime tick — an orchestration decision). Same precedent as TASK-026 `Unit.resetMovement()` shipping the verb without the Start-phase hook, and TASK-024's `TurnManager.on()` shipping hooks without serializing them.
  - Threshold ladder exposed as a **scalar seam** — the `ChimesLedger` constructor accepts an `init.thresholds` override, defaulting to `LIBERTY_CHIMES_THRESHOLDS`. Reason: TASK-066 (Archive Charter pool + pick-2 UI) will own how many charters exist and may re-tune the ladder or make it faction-specific; forcing it to monkey-patch a global would break the primitive's "same inputs produce same outputs" guarantee. By passing the ladder through construction, the charter task (and any future campaign-modifier task) gets to thread a ladder without touching the primitive. Test coverage explicitly validates the override path at short `[10, 25, 60]` and `[10, 20]` ladders so the primitive's behaviour under override is pinned, not just under the default. Matches the `scaleTileYield(base, multiplier)` + `buildingEffort` scalar-seam precedent from TASK-042 / TASK-041.
  - Threshold-crossing detection iterates the ladder in-order with a single early `break` when `this._chimes < threshold`, and stops checking further thresholds — this is a correctness guarantee, not just an optimisation. Because the ladder is validated strictly ascending at construction time (non-ascending ladders throw `RangeError`), once the first un-crossed threshold fails the `< threshold` check every subsequent one also fails. Stopping there means a huge accumulate (`amount = 10_000`) against a four-threshold ladder is O(number-of-newly-crossed) not O(ladder-length × log(ladder-length)), but more importantly the semantics are self-documenting: the code reads "walk ascending, skip already-crossed, stop at first un-crossed we haven't met." A descending-ladder bug would break this — hence the construction-time strict-ascending assert.
  - Pending-event queue is a FIFO of `CouncilEvent` objects, not a single "next threshold" integer. Reason: a single tick can cross *multiple* thresholds (the test asserts this: `accumulate(400)` against `[50, 150, 300]` fires three events). The player needs to resolve the three Councils one at a time — a single integer field couldn't represent the pending trio, and a "max crossed so far" approach would lose the mid-tier Councils entirely. FIFO matches how Colonization handled multi-turn events, and matches the deferred-across-turns semantics named in the task description ("Events defer until player returns from busy actions"). Separating `_crossed` (a Set that says "this threshold has been seen") from `_pending` (an array that says "these are un-consumed deferrals") means `consumeNextEvent()` drains the queue without reopening thresholds — a consumed event does not become re-firable, it becomes resolved. Test pins this: `crossedThresholds` survives `consumeNextEvent()` + a follow-up `accumulate(100)` fires no new event.
  - `pendingEvents` getter returns a **per-access copy** (`this._pending.map(ev => ({ threshold: ev.threshold }))`), not the internal array. Callers commonly want to "peek the queue" without popping, and reading the array reference directly would hand out a mutable view — a buggy caller could splice the queue without going through `consumeNextEvent()`, silently desyncing `_crossed`. The copy costs O(pending-length) which is bounded by the ladder size (4 in the MVP), and its existence is asserted in a test that mutates the returned array and verifies the ledger is unchanged. Same defensive-copy shape as `Colony.crew` / `Colony.buildings` (sorted-copy getter) and `HomePort.netVolume()` (returns a number, not an internal cell reference).
  - Save-format arrays sorted for byte-parity determinism — `toJSON.crossed` is emitted ascending (from the naturally-ordered threshold values), `pending` is emitted in FIFO order (which happens to be ascending because crossings always enqueue in ladder order), and `thresholds` is emitted in its construction order (which is ascending per construction-time validation). Two ledgers that reached the same `(chimes, crossed, pending)` state via *different accumulate sequences* produce identical `JSON.stringify` output — explicitly pinned by a test that accumulates `[5, 5, 10]` in one ledger vs `[20]` in another against the same `[10, 20]` ladder and asserts the two `JSON.stringify(toJSON())` outputs match. Matches the `CargoHold.toJSON()` + `Colony.toJSON()` determinism precedent from TASK-034 / TASK-038.
  - Content-side `chimes-flavour.ts` mirrors the threshold ladder next to narrative copy (heading + summary + preamble + register) and pins the core numbers via sibling drift guards in each package — same duplication-with-drift-test shape as `PlayableFactionId` (TASK-063) and `ShipClassEntry.baseMovement` (TASK-027). Reason: a tooltip or Council modal wants heading + preamble + threshold number in one read, and content can't import core per the dependency-direction rule, so the threshold number duplicates; two sibling tests cheaply guard against drift. Canon register uses salt-and-rum for thresholds 50/150/500 and eldritch for 300 (the Kraken-witnessed council) — the register variance is intentional and test-asserted, so a casual edit that re-registers all four to the same tone would fail the "third threshold opens eldritch" assertion.
  - `CouncilEvent` kept intentionally minimal — `{ readonly threshold: number }` — even though a future Council UI will want heading + charter-slot count + preamble. Per CLAUDE.md's "Trim consumer-specific fields off save-format registries" rule, the heading / preamble live in content (`COUNCIL_THRESHOLD_FLAVOURS`) and the Council UI's render path looks them up via `getCouncilThresholdFlavour(event.threshold)`. Charter-slot count is a TASK-066 concern that depends on Archive-Charter pool size — baking it into `CouncilEvent` now would force a save-format migration when TASK-066 picks its number. The only invariant `CouncilEvent` needs for save-format parity *is* the threshold, because the threshold is the primary key the consumer uses to look up everything else.
