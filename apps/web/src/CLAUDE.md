# apps/web/src — pattern rules

Pattern rules covering React HUD / screens and Zustand store discipline for
`apps/web/src/`. Loads whenever Claude reads a file under this directory.
Phaser-specific rules live in the nested `apps/web/src/game/CLAUDE.md`.

## Zustand store patterns

Multi-slice atomic writes and user-input-boundary clamping for the web
client's `apps/web/src/store/game.ts`.

- **Multi-slice zustand mutations commit in a single `set(...)` call.** When a store action mutates two or more independent slices that must agree afterwards (a player's `units` roster + a `homePorts` faction map; a `units` roster + a `colonies` faction map; a `homePorts` map + a `units` roster + a `coinLedger`), produce *one* `set((state) => ({ ...newSlices }))` call rather than chaining a slice-per-action. Mid-action a derived figure (`homePort.netVolume(resource)`, `unit.cargo.qty(resource)`, future `coinLedger[faction]`) can transiently disagree with the shipped state — that's fine — but external observers (other components subscribed to either slice) must never see a "ship lost 3 timber but port netVolume still 0" partial-write. Canonical examples: `commitTrade` in `apps/web/src/store/game.ts` writes new `units[shipIdx].cargo` and new `homePorts[factionId].netVolume` slices in one `set` (TASK-048); `commitCargoTransfer` writes new `units[shipIdx].cargo` and new `colonies[colonyId].cargo` slices in one `set` (TASK-049). Reason: zustand publishes a render between every `set`; a re-render that observes one updated slice but not the other paints the partial state and a Convert / cancel mid-frame can permanently diverge them. Resist splitting into convenience actions ("update port" + "update cargo") even when each side has a natural verb name — the atomicity is the rule, not an optimisation. New store actions that touch ≥ 2 slices should follow the same shape, with a test asserting both slices change together (`expect(state.units[0].cargo.timber).toBe(1)` + `expect(state.homePorts.otk.netVolume('timber')).toBe(-1)` after one action call).
- **Clamp-and-skip out-of-range qty at the zustand store boundary.** Store actions that consume slider/input values (a buy/sell qty, a load/unload qty, a queue-item priority) sit at the *user-input boundary* — even when the UI's slider bounds enforce the valid range, a future caller (hotkey, AI orchestrator, save-reload mid-action, replay log) can pass an out-of-range value. Default to per-line clamping + silent skip + continue with the rest of the cart, *not* throw or no-op the whole batch. The slider already prevents the bad value through the UI; the guard is *defence in depth*, not a contract violation that should crash the app. Canonical examples: `commitTrade` in `apps/web/src/store/game.ts` skips a sell line when `sellQty > shipCargo[resource]` (TASK-048); `commitCargoTransfer` clamps a load to `min(qty, colonyStock)` and an unload to `min(-qty, shipCargo)` per line (TASK-049). Reasons over throw: (a) the UI can't usefully react to an exception from a "submit cart" button — a partial clamp leaves the player one click closer to a successful trade, an exception leaves the cart broken; (b) one malformed line shouldn't kill the other nine; (c) silent-skip composes with the multi-slice atomic-`set` rule above — the action still produces a single render with a coherent (clamped) state. Reasons over no-op-the-whole-batch: same — the user's intent is "do as much of this cart as is legal," and a 90% trade is closer to that than a 0% trade. This is one of the few places the Tier 1 "trust internal code and framework guarantees" rule relaxes — the store boundary is user-input adjacent, so cheap edge guards earn their keep.

## UI screen patterns

React HUD/menu conventions: CSS Modules, click-through overlay scoping,
terminal-vs-overlay screen taxonomy, slice-driven event-modal mounting,
heraldic inline SVG, and the pure-sibling module pattern for non-trivial
React component logic.

- **HUD styling uses CSS Modules.** React components that overlay the Phaser canvas (HUD, menus, tooltips) colocate a `*.module.css` file next to the component and import it as `styles`. Vite handles CSS Modules out of the box — no Tailwind or runtime CSS-in-JS. Top-level layout (`App.css`) stays as plain global CSS; component-scoped styles live in modules to prevent class-name collisions as the HUD grows. Tests identify components via `data-testid` rather than class names so CSS-Module hashing stays transparent.
- **HUD overlays stay click-through where empty.** Any React layer mounted over the Phaser canvas (HUD chrome, fog overlay, pause menu, tooltips) wraps its root in a container that owns positioning + the inherited `pointer-events: none`, with `> *` flipping `pointer-events: auto` on the actual interactive children. Pattern (see `apps/web/src/hud/Hud.module.css`): a `.game-stage` parent with `position: relative`, then `.hud { position: absolute; inset: 0; pointer-events: none }` and `.hud > * { pointer-events: auto }`. Without this, an absolutely-positioned overlay that covers the canvas eats every Phaser map click — ergonomically broken even when the overlay is mostly transparent. New overlay components must keep to this container pattern; tests should still locate elements via `data-testid` since the wrapper class is hashed by CSS Modules.
- **Top-level screen routing lives in the zustand store as a literal-union.** App-level navigation between `'menu' | 'faction-select' | 'game'` (and future stops: `'pause'`, `'colony'`, `'diplomacy'`, `'codex'`, `'prologue'`, `'game-over'`) is a single `screen` slice in `apps/web/src/store/game.ts` rather than a routing library. `App.tsx` is a switch-on-`screen` dispatcher. Tests set the screen state directly in `beforeEach` (`useGameStore.setState({ screen: 'game' })`) instead of simulating navigation events — the test surface stays small and resilient to UI re-shuffles. Reasons over react-router: no extra dep, Phaser's canvas state is independent of URL state anyway (deep-linking into mid-game is an explicit non-goal), and the menu-vs-game split needs its own conditional anyway since they share no chrome. New screens add a literal to the union + a case to `App.tsx` + a `setScreen('<new>')` call from the originating UI; tests must split into per-screen suites (one `beforeEach` per terminal `screen` value) to avoid mounting the wrong root component. **Two sub-shapes exist: *terminal* screens vs *overlay* screens.** Terminal screens (`'menu'`, `'faction-select'`, `'game'`, `'game-over'`, `'prologue'`) each render their own root component; switching between them unmounts the previous root, so Phaser state does not survive `'game' → 'menu'` without explicit save. *Overlay* screens (`'pause'` today; `'colony'`, `'diplomacy'`, `'codex'` on the roadmap) render *on top of* the `'game'` root while the Phaser `GameScene` stays mounted beneath — because the player expects Esc-then-resume to continue the same battle, not restart it, and because a destroyed Phaser game is expensive to reconstruct. Mechanics: `App.tsx` dispatches the game-stage components (`<GameCanvas />` + `<Hud />`) for every member of the overlay family *plus* `'game'`, then layers the overlay component conditionally (`screen === 'pause' && <PauseOverlay />`). Canonical example: `apps/web/src/pause/PauseOverlay.tsx` (TASK-016); the overlay emits the `'game:pause'` / `'game:resume'` bus events on mount/unmount so the Phaser scene is suspended without the React tree needing to know about the game instance. Tests for overlay screens therefore assert *both* the terminal-root presence AND the overlay layer (see `App.test.tsx` pause suite asserting `getByTestId('hud')` + `getByTestId('pause-overlay')` co-mount). New overlay screens follow the same shape: add the literal, add the `App.tsx` dispatcher entry to the overlay family, mount-emit the pause bus event pair on overlay lifecycle.
- **Slice-driven self-mounting overlays for "happens-during-an-action" events, NOT new `Screen` literals.** Transient event modals that fire at any point during gameplay (combat resolved, rumour revealed, future: turn-event banner, treaty-accepted confetti, discovery cutscene) mount as a single `<Component />` inside `App.tsx`'s game-stage block *unconditionally* and branch internally on a dedicated store slice (`combatOutcome: CombatOutcome | null`, `rumourReveal: RumourOutcome | null`) — returning `null` when the slice is null. They do NOT add a new literal to the `Screen` union and they do NOT gate on `screen === '…'`. Canonical examples: `apps/web/src/combat/CombatOverlay.tsx` gated on `combatOutcome !== null` (TASK-055), `apps/web/src/hud/RumourRevealModal.tsx` gated on `rumourReveal !== null` (earlier EPIC-04 task); both live inside the `.game-stage` block in `App.tsx` alongside `<Hud />` and the screen-literal-dispatched overlays. Why a slice + unconditional mount beats a `Screen` literal for this family: (a) these events fire while the player is on *whatever* screen they were on — mid-game, mid-colony, mid-trade — and hard-binding them to a `screen === 'combat'` dispatch would force the orchestrator to remember the previous screen value and route back on dismiss, which is redundant with the fact that an overlay that `return null`s leaves the underlying screen mounted and interactive; (b) a `Screen` literal implies a mutual-exclusion relationship with other screens ("at most one is active"), but a combat-outcome reveal should NOT block the pause overlay or a colony screen-click — different slices can coexist, different screen literals can't; (c) dismiss is a store action (`dismissCombatOutcome`, `dismissRumourReveal`) that clears the slice, leaving the `screen` value untouched. Distinct from the terminal/overlay-*screen* dichotomy above (which governs player-chosen UI-navigation chrome like Pause, Colony, Diplomacy — the player *opens* those via a button or Esc; these modals appear *unbidden* as a consequence of a gameplay action). New event-modal components follow the same shape: add a nullable slice + show/dismiss actions to the store, add `<YourOverlay />` to `App.tsx`'s game-stage block unconditionally, guard the render inside the component with `if (!slice) return null`, write tests that set the slice via `useGameStore.setState(…)` and assert mount/unmount on slice change — not tests that drive the `screen` value.
- **Heraldic / crest visuals are inline SVG until the art epic ships.** Faction crests, OTK heraldry, and other emblematic chrome render as inline `<svg>` inside the React component during the pre-art-epic MVP — see `apps/web/src/menu/MainMenu.tsx` (paired-dragons motto) and `apps/web/src/menu/FactionSelect.tsx` (per-faction shield + sigil). Reasons: zero asset-pipeline pressure (no atlas regen, no preload phase, no Capacitor sync gymnastics), one PR per heraldic surface, and an obvious swap-point later when the pixel-art crests land — replace the `<svg>` body without touching the React tree. Constraints: scope every visual with a stable `data-testid` (e.g. `faction-crest-otk`) so tests pin structure-not-style and survive the future raster swap; mirror duplicates with `transform="scale(-1 1)"` rather than copy-pasting paths; any SVG that would push a single-component file past ~150 lines moves into a sibling `*.svg.tsx` partial. Inline SVG is not the long-term answer for the in-game tile atlas (which goes through the source/packed/served pipeline) — it's specifically for menu/HUD heraldic chrome.
- **Pure-sibling pattern also applies to non-trivial React component logic.** The Phaser-isolation rule (see `apps/web/src/game/CLAUDE.md`) is the *strict* form (driven by jsdom flakes); the same module shape is the recommended *ergonomic* form for React surfaces with non-trivial pure logic — the row/cell math behind a slider grid, the per-line totals behind a multi-row form, the predicate-set behind a "should this control render?" gate. Move that logic into a sibling `*-math.ts` (or `-rows.ts`, `-eligibility.ts`) under the component's directory and have the component import + render. Canonical examples: `apps/web/src/trade/trade-math.ts` (`computeTradeLine` / `computeTradeTotals`) tested directly with plain `HomePort` instances + `TradeScreen.tsx` untested for arithmetic, only DOM (TASK-048); `apps/web/src/transfer/transfer-math.ts` (`computeTransferRows`) tested directly with plain JSON snapshots + `CargoTransferScreen.tsx` untested for arithmetic (TASK-049). React tests *can* exercise this logic through the component (jsdom doesn't trip on React-only modules), but the indirection is real testing cost: a slider drag → re-render → DOM-text assertion is slower to write, slower to fail, and noisier to read than a direct call to a pure function with explicit inputs. Use the sibling whenever the logic survives without the component (alphabetical-key sort, qty-sign branch selection, currency arithmetic, predicate composition) — the component file then shrinks to wiring + JSX. Skip it when the logic is single-expression and only ever the JSX `{cond ? a : b}` ternary; trivial-on-render code doesn't need a module.
