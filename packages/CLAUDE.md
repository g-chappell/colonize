# Packages — pattern rules

Pattern rules covering `packages/core`, `packages/content`, and `packages/shared`:
registry design + save-format serialisation machinery. This file loads whenever
Claude reads a file under `packages/`.

## Registry patterns

Const-object registries (BuildingType, UnitType, ShipClass, …), core-vs-
content split of rule-relevant stats, and test discipline for balance-
tunable registries. The string-literal shape for save-format-bound
registries lives in the root `CLAUDE.md` under "TypeScript patterns".

- **Relational invariants over literal numbers in balance-tunable registry tests.** When a content-side or core registry holds numbers that product / design will rebalance (campaign turn-counts, ship-class stat blocks, profession multipliers, reputation thresholds, wave escalations, council-threshold ladders), tests pin the *design-intent relationships* — monotonicity, ordering, uniqueness, non-empty counts, cross-tier escalation — rather than the literal values. A test that asserts `pacified.turnsRequired === 12` breaks the moment a balance pass trims pacified to 10 turns; a test that asserts `pacified.turnsRequired < standard.turnsRequired < brutal.turnsRequired` survives every rebalance that preserves the design intent. Canonical examples: `packages/content/src/concord-campaign.test.ts` (TASK-070) asserts pacified < standard < brutal on both `turnsRequired` AND total deployed-unit count, `spawnTurn` non-decreasing per tier, every `spawnTurn < turnsRequired`, every ship/ground id resolves via `isShipClassId`/`isGroundClassId`, plus non-empty copy fields — without locking any literal. `packages/content/src/units.test.ts` (TASK-027) pins sloop/privateer fastest `baseMovement`, ship-of-the-line slowest, hull ascending sloop → ship-of-the-line. `packages/core/src/chimes/chimes-registry.test.ts` (TASK-065) locks threshold-ladder ordering + cardinality rather than the exact values. `packages/core/src/charter/charter-registry.test.ts` (TASK-066) pins count=20 + hand-size=2 + non-empty copy + unique ids rather than individual charter effects. Tests that DO lock literals should be few and deliberate — save-format magic numbers, TurnPhase member count pinned against wire-compat migrations, `CORE_VERSION` — and named so the reason-for-pinning is visible (`version_bump_breaks_wire_compat`, not `chapel_costs_3_timber`). This is the testing-pattern counterpart to the "Rule-relevant stats live in `@colonize/core`" rule (which already prescribes an ordering-invariant test for the duplicated-number case); applying it broadly saves a fix-cycle per balance tweak and keeps CI feedback useful during the MVP-balance phase.
- **Rule-relevant stats live in `@colonize/core`; descriptive / flavour stats live in `@colonize/content`.** A numeric attribute that the engine *reads to resolve a rule* (movement points, hull HP, upkeep cost, resource yield, combat stats) belongs next to its registry in `packages/core` — that's what `Unit.resetMovement()` reads out of `getUnitTypeDefinition(…).baseMovement` in TASK-026. A descriptive attribute (display name, tooltip copy, lore description, UI icon hint) belongs in `packages/content`. The canonical split is TASK-027's ship registry: `baseMovement` lives in `packages/core/src/unit/unit-type.ts` *and* on `ShipClassEntry` in `packages/content/src/units.ts`, because a single tooltip read wants the full stat block. Prefer the duplicated number over a `content → core` import: the dependency-direction rule (`web/mobile/server → core/shared/content`, never the other way) forbids content importing core, and a new cross-workspace edge to save two lines loses against the Tier 1 "don't design for hypothetical future requirements" rule. When duplicating, pin consistency with a test in the content workspace that asserts the internal ordering invariant (sloop/privateer fastest `baseMovement`, ship-of-the-line slowest; hull ascending sloop → ship-of-the-line) so drift between the two copies fails loudly in CI. Future stats likely to follow this split: resource base yields, building upkeep, profession output multipliers, combat attack/defense, ship cargo capacity.
- **Trim consumer-specific fields off save-format registries.** When a task description enumerates a registry field whose semantic meaning lives in a consumer system that doesn't yet exist (a building's gameplay `effect` string, a blueprint's `unlocked` flag, a tile's descriptive subtype used only by a single future yield-system), drop the field rather than invent a placeholder. The consumer system when it lands reads the registry id through its own lookup (`hasBuilding(BuildingType.Shipyard)`, `legendaryFleet.isBuildable(id)`, `getTileYield(TileType.Island)`) and applies its own semantics — no encoded hook on the registry struct is required for consumers to attach behaviour. Canonical examples: `BuildingDefinition` in `packages/core/src/building/building-type.ts` deliberately omitted `effect: string` even though TASK-040's description mentioned "Shipyard enables ship construction" / "Chapel grants Talisman blessings" — the shipyard construction flow and the rumour-blessing system each query `hasBuilding(type)` through their own entry point (TASK-040). `LegendaryShipSlot` dropped the static `unlocked: false` field even though an earlier draft carried it — per-faction runtime state owned by `LegendaryFleet.discovered` / `redeemed` sets is the real source of truth (TASK-035). `TileType` was NOT expanded to add `fishing-waters` / `kelp-forest` / `coastal-grove` members even though TASK-042's description used those names — those tile-kind descriptors were interpreted as subtypes of the existing coarse `Ocean` / `Island` members, with the yield vector in `getTileYield` doing the per-kind work any consumer needed (TASK-042). Reasons over adding the speculative field: (a) it becomes either dead flavour duplicating `summary` / `description` or a poorly-typed union trying to encode every consumer's entry point before any consumer exists; (b) every new save-format member forces a migration for every wire-format change the consumer later wants, so adding one speculatively pays the migration cost twice; (c) the consumer's semantics belong with the consumer, where the change cadence is local. Distinct from "Ship the entity's primitive; leave iteration / scheduling to the task that owns the collection" (which governs *when / to whom* a verb fires) — this is about *which fields live on the registry's struct* versus *which fields live on the consumer's own state*.

## Serialisation patterns

Save-format machinery: `toJSON` / `fromJSON` discipline, determinism guards,
threshold-accumulator shape, defensive-copy getters. Any primitive whose
state round-trips through the save format falls under these rules.

- **Save format:** `packages/core` exports `serialize/deserialize` for game state. Same format used for local save and cloud save. Version with an integer; add migrations in `packages/core/migrations/`.
- **Map/Set-backed save-format emitters sort entries in `toJSON` for byte-parity determinism.** Any primitive whose internal state uses `Map` or `Set` and whose JSON emission is part of the save format must emit entries in a stable sorted order (alphabetical for string keys, row-major index order for coordinate keys, insertion-order-of-sorted-list for array snapshots). Without the sort, two instances that received the same mutations in a different order produce different JSON — breaking save-file replay parity across runs, machines, and cross-version reloads even though the two instances are logically equal. JS `Set` and `Map` preserve insertion order, and that insertion order leaks straight into a naive `JSON.stringify`. Canonical examples: `CargoHold.toJSON()` in `packages/core/src/cargo/cargo-hold.ts` sorts resource ids alphabetically and artifact ids alphabetically (TASK-034); `FactionVisibility.toJSON()` emits cells in row-major index order; `DirectionLayer.toJSON()` emits sparse entries sorted by row-major index (TASK-029); `Colony.toJSON()` in `packages/core/src/colony/colony.ts` sorts the crew + building id snapshots alphabetically (TASK-038). The sort is cheap (small-N, short kebab-case strings) and the determinism guard is a one-liner test: two differently-ordered mutation sequences that end in the same logical state must produce identical `JSON.stringify(x.toJSON())`. Applies to every future save-format primitive that backs its state with a `Set` or `Map` — ResourceStockpile, FactionRegister, ReputationLedger, etc. — write the determinism test alongside the primitive's `toJSON`, not as a follow-up when a cross-machine replay diverges.
- **FIFO-queue + re-fire-guard `Set` pattern for threshold-crossing accumulators.** When a primitive accumulates a scalar value and needs to fire a distinct event every time the value newly crosses one of N fixed thresholds, use a sibling `_pending: Event[]` array (FIFO drain queue) + `_crossed: Set<number>` (re-fire suppression) pair — NOT a recompute-crossings-from-scratch scan on each mutation. Pattern: on each `accumulate` / `tick` / `apply` call, walk the threshold ladder ascending; for each threshold `t` where `prev < t <= next` AND `!_crossed.has(t)`, push an event to `_pending` and add `t` to `_crossed`. Drain via `consumeNextEvent()` (FIFO shift). Save-format emits both in sorted order for byte-parity (the Map/Set-sorted-toJSON rule). Canonical examples: `ChimesLedger` in `packages/core/src/chimes/chimes-ledger.ts` (TASK-065 — Council-event thresholds on Liberty Chimes accumulation) and `ConcordTensionMeter` in `packages/core/src/concord/tension-meter.ts` (TASK-067 — Ultimatum-event thresholds on tension accumulation) — both ship the same ~20-line shape independently. TASK-067's lesson explicitly cites "mirrored the ChimesLedger shape rather than generalizing" as the right call. Prefer a mirrored sibling over extracting a generic `ThresholdAccumulator<Event>` base class: the two consumers already diverge on event-payload shape (`CouncilEvent` is a bare `{threshold}`; `ConcordUltimatumEvent` carries a reason discriminator), the third consumer (a future reputation / fame / trade-volume / infamy ledger under EPIC-07 / 09) will diverge further, and generic extraction buys ~10 shared lines at the cost of a three-generic signature every caller has to parametrise. Reasons over a recompute-each-time scan: (a) walking the `_crossed` set is O(1) per threshold lookup vs O(N) scan of the full ladder, (b) the `_pending` queue naturally supports the orchestrator's "drain one event per turn" pacing without re-deriving which events are still unconsumed, (c) the save format round-trips both fields so crash-recovery mid-drain resumes without re-firing already-consumed events. Tests should cover: single-tick multi-threshold burst (crossing three thresholds in one `accumulate` enqueues three events FIFO), re-fire suppression (re-crossing a threshold after decrement+reincrement does not re-enqueue), and `toJSON` round-trip parity (deserialised instance replays the same future events).
- **Per-access defensive copies on pending-state getters.** Primitive getters that expose queued or deferred internal collections (FIFO event queues, `Set`-backed crossed-threshold indices, pending-wave schedules, sorted ledger snapshots) return a *per-access copy* of the internal state, never a live reference. Without the copy, the orchestrator that walks the list to "consume" an entry can mutate the internal storage by accident (a `.pop()` to dequeue, a `.splice()` to reorder, a `.shift()` to drain) and permanently corrupt the primitive's invariants — the bug is silent at the call site and only surfaces the next time the primitive emits state. Canonical examples: `ChimesLedger.pendingEvents` in `packages/core/src/chimes/chimes-ledger.ts` returns `this._pending.map((ev) => ({ threshold: ev.threshold }))` — a fresh array of fresh event records (TASK-065); `ConcordTensionMeter.pendingEvents` in `packages/core/src/concord/tension-meter.ts` mirrors the same shape (TASK-067); `ConcordFleetCampaign.pendingWaves()` in `packages/core/src/concord/concord-fleet-campaign.ts` deep-copies each wave's `ships` and `groundTroops` arrays before returning (TASK-070); `ConcordFleetCampaign.waves` getter copies every wave and its arrays for the same reason; `ConcordFleetCampaign.spawnedWaveIndices` sorts-and-spreads a `Set` into a fresh array. Applies symmetrically to `toJSON()` emission — the Map/Set-backed sort-then-spread rule above is the *wire-format* case; this rule is the *in-memory API* case. The copy is cheap (small-N, short kebab-case strings or plain-object records) and the discipline is a one-line test: "mutating the returned collection must not change the next getter call's result". Write the defence alongside the getter, not as a follow-up after an orchestrator accidentally mutates it.
