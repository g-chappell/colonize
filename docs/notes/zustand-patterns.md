# Zustand store patterns

Multi-slice atomic writes and user-input-boundary clamping for the web
client's `apps/web/src/store/game.ts`.

- **Multi-slice zustand mutations commit in a single `set(...)` call.** When a store action mutates two or more independent slices that must agree afterwards (a player's `units` roster + a `homePorts` faction map; a `units` roster + a `colonies` faction map; a `homePorts` map + a `units` roster + a `coinLedger`), produce *one* `set((state) => ({ ...newSlices }))` call rather than chaining a slice-per-action. Mid-action a derived figure (`homePort.netVolume(resource)`, `unit.cargo.qty(resource)`, future `coinLedger[faction]`) can transiently disagree with the shipped state — that's fine — but external observers (other components subscribed to either slice) must never see a "ship lost 3 timber but port netVolume still 0" partial-write. Canonical examples: `commitTrade` in `apps/web/src/store/game.ts` writes new `units[shipIdx].cargo` and new `homePorts[factionId].netVolume` slices in one `set` (TASK-048); `commitCargoTransfer` writes new `units[shipIdx].cargo` and new `colonies[colonyId].cargo` slices in one `set` (TASK-049). Reason: zustand publishes a render between every `set`; a re-render that observes one updated slice but not the other paints the partial state and a Convert / cancel mid-frame can permanently diverge them. Resist splitting into convenience actions ("update port" + "update cargo") even when each side has a natural verb name — the atomicity is the rule, not an optimisation. New store actions that touch ≥ 2 slices should follow the same shape, with a test asserting both slices change together (`expect(state.units[0].cargo.timber).toBe(1)` + `expect(state.homePorts.otk.netVolume('timber')).toBe(-1)` after one action call).
- **Clamp-and-skip out-of-range qty at the zustand store boundary.** Store actions that consume slider/input values (a buy/sell qty, a load/unload qty, a queue-item priority) sit at the *user-input boundary* — even when the UI's slider bounds enforce the valid range, a future caller (hotkey, AI orchestrator, save-reload mid-action, replay log) can pass an out-of-range value. Default to per-line clamping + silent skip + continue with the rest of the cart, *not* throw or no-op the whole batch. The slider already prevents the bad value through the UI; the guard is *defence in depth*, not a contract violation that should crash the app. Canonical examples: `commitTrade` in `apps/web/src/store/game.ts` skips a sell line when `sellQty > shipCargo[resource]` (TASK-048); `commitCargoTransfer` clamps a load to `min(qty, colonyStock)` and an unload to `min(-qty, shipCargo)` per line (TASK-049). Reasons over throw: (a) the UI can't usefully react to an exception from a "submit cart" button — a partial clamp leaves the player one click closer to a successful trade, an exception leaves the cart broken; (b) one malformed line shouldn't kill the other nine; (c) silent-skip composes with the multi-slice atomic-`set` rule above — the action still produces a single render with a coherent (clamped) state. Reasons over no-op-the-whole-batch: same — the user's intent is "do as much of this cart as is legal," and a 90% trade is closer to that than a 0% trade. This is one of the few places the Tier 1 "trust internal code and framework guarantees" rule relaxes — the store boundary is user-input adjacent, so cheap edge guards earn their keep.
